# 020_CLAUDE_CONFIG_DIR による双方向パス互換性の実現

## ステータス

承認済み

## 日付

2026-03-09

## コンテキスト

[ADR 019](019_DevContainerでのClaude%20Code設定共有と動的パス解決.md) で、`remoteEnv` と動的シンボリックリンクにより Plugin Marketplace のパス問題を解決した。しかし、この方式はホスト→コンテナ方向のパス解決のみを実現しており、逆方向の問題が残っていた。

### 現状の問題点

- **コンテナでインストールしたプラグインがホストで使用できない**: コンテナ内で Plugin Marketplace からプラグインを追加すると、`known_marketplaces.json` の `installLocation` がコンテナのパス（`/home/vscode/.claude/plugins/...`）で記録される。ホスト側にはこのパスが存在しないため、プラグインを読み込めない
- **ADR 019 のシンボリックリンク方式は片方向**: ホストのパス（`/home/alice/.claude/...`）をコンテナ内で解決する仕組みであり、コンテナのパスをホスト側で解決する仕組みはない

### 制約条件

- ADR 019 のシンボリックリンク方式は引き続きホスト→コンテナ方向のパス解決に必要
- バインドマウント先は `/home/vscode/.claude` のまま維持する（マウント先をホストのパスに変更する場合、コンテナ内にホストのホームディレクトリと同じパスを事前に作成し権限を設定する必要があり、Dockerfile にホスト固有の情報を持ち込むことになるため避ける）
- ホスト側の設定ファイルに副作用を与えてはならない

## 決定

`devcontainer.json` の `remoteEnv` に `CLAUDE_CONFIG_DIR` 環境変数を追加し、コンテナ内の Claude Code が設定ディレクトリとしてホストと同じ絶対パスを使用するようにする。既存のシンボリックリンク方式と組み合わせることで、双方向のパス互換性を実現する。

### 実装方針

1. **`remoteEnv` に `CLAUDE_CONFIG_DIR` を追加**: `${localEnv:HOME}/.claude` を設定し、コンテナ内の Claude Code がホストと同じパスで設定ディレクトリを参照するようにする
2. **既存のシンボリックリンク方式を維持**: `setup-claude-paths.sh` による `HOST_HOME` → `HOME` のシンボリックリンクにより、`CLAUDE_CONFIG_DIR` が指すパス（`/home/alice/.claude`）がコンテナ内の実ファイル（`/home/vscode/.claude`）に解決される
3. **バインドマウント設定は変更しない**: `target=/home/vscode/.claude` のまま維持する

### パス解決の仕組み

| 方向 | 解決手段 | 動作 |
|---|---|---|
| ホスト→コンテナ | シンボリックリンク（ADR 019） | ホストのパス `/home/alice/.claude/...` → シンボリックリンク経由で `/home/vscode/.claude/...` に解決 |
| コンテナ→ホスト | `CLAUDE_CONFIG_DIR`（本ADR） | コンテナ内の Claude Code が `/home/alice/.claude` をベースパスとして使用 → プラグインがホストと同じパスで記録される |

### `devcontainer.json` の変更

```json
{
  "remoteEnv": {
    "HOST_HOME": "${localEnv:HOME}",
    "CLAUDE_CONFIG_DIR": "${localEnv:HOME}/.claude"
  }
}
```

## 結果

### ポジティブな影響

1. **双方向のパス互換性が実現される**
   - ホストでインストールしたプラグインはシンボリックリンク経由でコンテナで使用可能（従来通り）
   - コンテナでインストールしたプラグインもホストと同じパスで記録されるため、ホストでも使用可能

2. **変更が最小限**
   - `remoteEnv` に1行追加するだけで実現できる
   - 既存のシンボリックリンク方式やバインドマウント設定に変更が不要

### ネガティブな影響・トレードオフ

1. **`CLAUDE_CONFIG_DIR` の仕様への依存**
   - Claude Code がこの環境変数を公式にサポートしていることが前提となる
   - 対策: 環境変数が無視された場合でも、既存のシンボリックリンク方式によりホスト→コンテナ方向は引き続き動作する

2. **ADR 019 のシンボリックリンクとの二重構成**
   - パス解決に2つの仕組みが共存するため、トラブルシューティング時に理解が必要
   - 対策: 各仕組みの役割分担を本ADRで明確に記録している

## 代替案

### 案1: バインドマウント先をホストのパスに変更する

**概要**: `target=${localEnv:HOME}/.claude` としてホストと同じパスにマウントし、`CLAUDE_CONFIG_DIR` も同じパスに設定する。シンボリックリンクが不要になる。

**メリット**:
- シンボリックリンクが不要になり、パス解決の仕組みが単純化される
- `setup-claude-paths.sh` を廃止できる

**デメリット**:
- マウント先の親ディレクトリ（例: `/home/alice`）が Docker により `root` 所有で自動作成される
- 権限調整のために `postCreateCommand` で `sudo chown` が必要になる可能性がある
- Dockerfile でホストのパスを事前準備しようとすると、ホスト固有の情報をイメージに持ち込むことになる

**却下理由**: マウントポイントの権限問題の検証が必要であり、既存のシンボリックリンク方式に `CLAUDE_CONFIG_DIR` を追加する方が確実で変更が少ない

### 案2: 現状維持（シンボリックリンクのみ）

**概要**: ADR 019 の方式のまま、コンテナ→ホスト方向のパス問題は許容する。

**メリット**:
- 追加の設定変更が不要
- 既に動作実績がある

**デメリット**:
- コンテナでインストールしたプラグインがホストで使用できない

**却下理由**: プラグインの追加・更新をホスト・コンテナどちらからでも行えることが開発体験として望ましい
