# 021_設計書のリポジトリ管理廃止

## ステータス

承認済み

## 日付

2026-07-18

## 信頼度

高 — 設計書とコードの乖離は本リポジトリで実際に発生している問題であり、保持対象（ADR・要件定義書・調査報告書）と削除対象（設計書）の区別は「コードから復元できるか」という明確な基準に基づく。

## 再評価条件

- 複数人での開発体制になり、実装前の設計合意プロセスとして設計書の共有・レビューが必要になった場合
- ADR・要件定義書・コードだけでは実装の全体像の把握に支障が出る事例が繰り返し発生した場合

## コンテキスト

`docs/` は主にサイト公開用コンテンツ原稿の管理の場であり（`docs/README.md`）、本ADRはそのうち開発文書（`adr/`・`specs/`・`design/`・`designs/`・`requirements/`・`reports/`）のみを対象とする。コンテンツ原稿・執筆ガイドライン（`published/`・`drafts/`・`templates/`・`guidelines/`）は対象外である。文書管理方針の決定は [001_ADR運用の導入](001_ADR運用の導入.md) の作成条件に明示されていないが、開発プロセスに関する決定を記録した先例（002・015・017）に倣い ADR として記録する。

開発文書には、ADR・要件定義書・設計書・機能仕様・運用ガイド・調査報告書が混在している。このうち設計書（機能設計書、機能仕様、実装ガイド）は「どう作るか」を記述した文書であり、実装が完了すればコードとテストがその内容を表現する。

### 現状の問題点

- **コードとの二重管理**: 設計書が記述する内容は実装完了後はコードが正となるが、文書としても残っているため、同じ情報を2箇所で管理する状態になっている
- **陳腐化と誤情報化**: 実装変更時に設計書は更新されておらず、コードと乖離した文書が正しい情報源であるかのように残り続けている。例: `work_hour_management_design.md` は2025-12-12以降未更新だが、その後実装には予算工数機能等が追加され文書に存在しない。`article_tags_feature.md` は [017_Article機能の廃止](017_Article機能の廃止.md) で対象機能が廃止された後も残存している
- **配置の乱立**: 開発文書が `docs/specs/`・`docs/design/`・`docs/designs/`・`docs/requirements/` に分散し、命名も不統一で（`docs/design/` にあるのは要件定義書、`docs/specs/` には設計書と要件定義書と調査報告書が同居）、どの文書が有効かが判別できない

### 制約条件

- コードから復元できない情報を失ってはならない。具体的には、決定の経緯と理由（ADR）、何を作るか・なぜ作るか（要件定義書）、技術検討の調査結果（調査報告書）
- 削除した設計書は Git 履歴に残るため、過去の記述が必要になった場合は履歴から参照できる

## 決定

設計書相当の文書をリポジトリで管理しないことを決定した。コードとテストが動作する仕様であり、設計書は実装前の一時的な成果物として扱う。

削除対象はファイル名パターンではなく文書種別で判定する。「どう作るか（How）」を記述した文書 — 機能設計書・機能仕様書・実装ガイド — が対象である。ADR・要件定義書・調査報告書は、コードに残らない情報を記録する文書として引き続きリポジトリで保持する。

### 実装方針

1. 保持する文書を用途別ディレクトリに整理する: `docs/adr/`（決定記録）、`docs/requirements/`（要件定義書）、`docs/reports/`（調査報告書）。現在 `docs/specs/`・`docs/design/` に混在している要件定義書・調査報告書は該当ディレクトリへ移動する
2. 既存の設計書相当の文書（機能設計書・機能仕様書・実装/運用ガイド）は削除する。事前の情報抽出（蒸留）は行わず、過去の内容が必要になった場合は Git 履歴を参照する
3. 削除後、`docs/specs/`・`docs/design/`・`docs/designs/` は廃止する
4. 今後の開発で実装前に設計書を作成する場合は一時的な作業成果物として扱い、リポジトリにはコミットしない。作業ファイルは実装完了後に破棄する

## 結果

### ポジティブな影響

1. **情報源の一本化**
   - 実装の仕様はコードとテストのみが正となり、乖離した文書を参照して誤った理解をするリスクがなくなる

2. **保守コストの削減**
   - 実装変更のたびに設計書を追従更新する作業（実際には行われず乖離の原因となっていた作業）が不要になる

3. **文書構成の明確化**
   - `docs/` 配下に残る開発文書はすべて「コードから復元できない情報」となり、各文書の役割と有効性が明確になる

### ネガティブな影響・トレードオフ

1. **実装全体像の把握材料が減る**
   - 実装済み機能の構造を俯瞰する文書がなくなる
   - 対策: 要件定義書（What）・ADR（Why）・コード（How）で補完する。俯瞰資料が必要な場合はその時点のコードから生成する

2. **設計書にのみ記録されていた情報へのアクセス性低下**
   - 設計書にのみ記録されていた設計判断・ドメイン知識がある場合、削除後は Git 履歴を辿らないと参照できない
   - 対策: 削除ファイルは Git 履歴に残る。参照する頻度が高い判断が見つかった場合は、その時点で個別に ADR 化する

## 代替案

### 案1: 現状維持（設計書をリポジトリに残し続ける）

**概要**: 既存の設計書をそのまま保持し、新規設計書も追加していく。

**メリット**:
- 作業が発生しない
- 実装当時の設計意図を文書の形で参照できる

**デメリット**:
- 二重管理と乖離が続き、陳腐化した文書が誤った情報源になり続ける
- 文書の乱立が解消されない

**却下理由**: 現に設計書の追従更新は行われておらず、乖離した文書を残すことは情報源として害になるため。

### 案2: 設計書を維持し、実装変更時の文書更新を義務化する

**概要**: 設計書を正式な文書として維持し、コード変更時に必ず設計書も更新するルールを設ける。

**メリット**:
- 実装と文書の整合が保たれれば、俯瞰資料として機能する

**デメリット**:
- コードが表現している情報を文書でも維持する二重管理コストが恒常的に発生する
- 更新徹底の仕組みがなく、ルールだけでは乖離の再発を防げない

**却下理由**: 二重管理コストに見合う便益がなく、コードから復元可能な情報のために保守負担を増やすことになるため。

### 案3: ADR・要件定義書を含む全開発文書を削除する

**概要**: コードのみを情報源とし、`docs/` 配下の開発文書をすべて削除する。管理対象は最小になる。

**却下理由**: 決定の経緯・理由（Why）と要件（What）はコードに現れず、失われると復元できない。保持と削除の基準は「コードから復元できるか」に置くべきである。

### 案4: 設計書をリポジトリ外（Wiki 等）へ移管する

**概要**: 設計書をリポジトリから出し、Wiki やナレッジベースで管理する。リポジトリはコードと必須文書のみになる。

**却下理由**: 問題の本質は配置場所ではなく二重管理であり、移管では解決しない。コードとの乖離がリポジトリ外でさらに検知しにくくなる。

### 案5: 削除前に Why を抽出（蒸留）してから削除する

**概要**: 各設計書からコードに現れない設計判断・ドメイン知識を抽出し、ADR・コードコメントへ移した上で削除する。

**却下理由**: 削除ファイルは Git 履歴から参照可能であり、全文書を対象とした抽出作業のコストに見合わない。必要が生じた時点で履歴から個別に参照・ADR 化すれば足りる。
