# 024_ADR版履歴の実現方式

## ステータス

承認済み

## 日付

2026-07-18

## 信頼度

高 — 記録項目は要件定義書4章と機能設計書 3.2.3 で確定済みであり、対象が単一モデル（`AdrManagement::Adr`）に限定されている。候補 gem の Rails 8.1 対応状況は RubyGems・公式 CHANGELOG で確認した（2026-07-18 時点）。

## 再評価条件

- 版履歴の対象がマスタ（クライアント・案件・プロジェクト）や他ドメインのモデルへ拡張される場合（汎用 gem の再検討価値が生じる）
- 過去の版への復元（revert）操作や版間差分の自動生成など、汎用バージョニング機能の要件が本格化した場合
- 記録項目が大幅に増減し、設計 3.2.3 のスナップショット方式自体を見直す場合

## コンテキスト

ADR 管理機能では、Coding Agent が MCP 経由で ADR を更新するため、意図しない書き換え・誤更新が起きた際に変更内容の確認と復元ができる必要がある（要件定義書4章。業務追跡目的であり、法令上の記録義務はない）。この版履歴の実現方式（自前テーブルか汎用 gem か）が未決定であり、機能設計書 3.2.3 でも記録内容のみ定義して方式は本 ADR に委ねられている。

### 記録要件（機能設計書 3.2.3）

- 対象 ADR・変更前の全項目スナップショット・変更されたフィールド・変更日時
- **変更種別**: 作成／更新／ステータス変更（置換の一体操作由来）／案件変更 の4種
- **更新経路（origin）**: MCP 経由なら OAuth アプリケーション名、Web 管理画面経由ならその旨とログイン中の管理者。MCP ツールは `server_context` 経由で更新経路を受け取る（機能設計書 6.3）
- ADR 本体の削除時は版履歴も併せて削除する（要件 4.4）

### 制約条件

- 環境は Rails 8.1（rails 8.1.3）/ SQLite3
- 版履歴の対象は `AdrManagement::Adr` の単一モデルのみ。マスタの変更は記録対象外（要件 4.2）
- 変更種別「案件変更」「ステータス変更（一体操作由来）」は CRUD イベントに一対一で対応しない業務イベントである
- 用語集で版履歴の英語名は `AdrRevision` 推奨とされているが、「gem に委ねる場合は再検討」と保留されており、本決定で確定する必要がある

## 決定

**自前の版テーブル（スナップショット方式）を採用することを決定した。** 汎用バージョニング gem（paper_trail / audited）は導入しない。

### 実装方針

1. モデル `AdrManagement::AdrRevision`（テーブル `adr_management_adr_revisions`）を作成し、機能設計書 3.2.3 の記録項目をそのまま列として持たせる
   - 対象 ADR への外部キー、変更前スナップショット（全項目を JSON で保持。作成時は作成であることが分かる形）、変更されたフィールド（JSON 配列）、変更種別（`created` / `updated` / `status_changed` / `engagement_changed` の enum）、更新経路（origin）、変更日時
2. 版の記録は ADR の登録・更新・置換の一体操作・案件変更の各操作のトランザクション内で明示的に行う（コールバックによる暗黙記録ではなく、操作の一部として記録する）
3. 更新経路は機能設計書 6.3 のとおり、MCP ツールが `server_context` から受け取った値を、Web 管理画面がログイン中の管理者情報を、それぞれ記録処理へ渡す
4. ADR 本体の削除時は関連の版履歴を併せて削除する（`dependent: :destroy` 相当）
5. 用語集の版履歴の英語名は `AdrRevision` で確定する（gem を採用しないため、保留条件が解消される）

## 結果

### ポジティブな影響

1. **記録項目と実装の一致**: 変更種別（案件変更・一体操作由来のステータス変更）や origin といったドメイン固有の項目を、gem のメタデータ機構への写像なしに素直な列として表現できる。設計書 3.2.3 の表とテーブル定義が一対一に対応し、参照画面（版履歴の変更フィールド明示）も列を読むだけで実装できる
2. **依存の不追加**: 汎用 gem への依存を持たないため、gem のメンテナンス停滞・Rails メジャーアップデート時の対応待ちのリスクを負わない
3. **記録タイミングの明示性**: 版の記録が各操作のトランザクションの一部として明示されるため、「どの操作で版が作られるか」がコードから追跡しやすく、テストも操作単位で書ける

### ネガティブな影響・トレードオフ

1. **記録処理の自作**: スナップショット生成・変更フィールド算出・記録漏れ防止を自前で実装・テストする必要がある。gem なら得られたはずの実績あるコールバック機構は使えない
   - 対策: 対象は単一モデル・記録契機は4操作に限定されており、実装範囲は小さい。記録漏れは受け入れ条件（作成・更新のたびに版履歴が記録されること）のテストで担保する
2. **汎用機能の不在**: 復元（revert）・版間差分・対象モデル追加は都度実装になる
   - 対策: 現要件の「復元」は過去の版の内容を参照して人が書き戻せれば足りる（参照者は開発者本人のみ、要件 4.5）。対象モデルが増えた場合は再評価条件に基づき gem 移行を再検討する

## 代替案

### 案2: paper_trail gem

**概要**: `has_paper_trail` で versions テーブルに変更履歴を自動記録する汎用 gem。最新 17.0.0（2025-10-24 リリース、activerecord >= 7.1）で Rails 8.1 サポートが CHANGELOG に明記されており、17.0.0 ではモデルごとのカスタム version クラス・テーブルにも対応。メンテナンスは活発。

**却下理由**: 技術的には成立するが、要件との適合で自前実装に劣る。(1) イベントは create/update/destroy が基本で、「ステータス変更（一体操作由来）」「案件変更」の業務的な変更種別はカスタムイベント名や metadata 列への写像が必要になり、設計 3.2.3 の項目と gem の標準列（`object` / `object_changes` / `event` / `whodunnit`）の対応関係という余分な翻訳層を保守することになる。(2) origin も `whodunnit` か metadata への詰め替えが必要。(3) 対象が単一モデルのみで、gem の主価値である「多数のモデルへの一括適用」を活かせず、依存追加と将来の陳腐化リスク（Rails メジャーアップデート時の gem 対応待ち）だけを負う。

### 案3: audited gem

**概要**: `audited` で audits テーブルに変更を記録する汎用 gem。最新 5.8.0（2024-11-08 リリース、activerecord >= 5.2, < 8.2）。

**却下理由**: (1) 記録されるのは差分（`audited_changes`）のみで、要件の「変更前の全項目スナップショット（過去の版として参照できること）」を満たすには差分の遡及再構成が必要になり、設計 3.2.3 と合わない。(2) 依存制約上 Rails 8.1（< 8.2）にはインストール可能だが、最終リリースは Rails 8.1 GA 前であり、README のサポート一覧も Rails 7.2 までで Rails 8 系の記載がない。リリース頻度も 2024-11 以降止まっており、メンテナンス状況の面で paper_trail に劣る。
