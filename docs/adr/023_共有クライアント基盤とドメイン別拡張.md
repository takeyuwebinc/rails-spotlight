# 023_共有クライアント基盤とドメイン別拡張

## ステータス

承認済み

## 日付

2026-07-18

## 信頼度

高 — ADR 管理機能の要件定義・機能設計の過程で代替案（工数管理ドメインへの参照・code 文字列連携・完全独立・単一モデル集約）を比較検討したうえでのユーザー決定である。共有カーネルの範囲を識別属性（code・name）に限定することで、ドメイン間結合のリスクを構造的に抑えている。

## 再評価条件

- 共有クライアントに識別属性以外の属性（住所・請求条件等のドメイン固有になりうる属性）を追加したくなった場合。共有カーネルの肥大化の兆候であり、どのドメインの拡張に置くべきかを再検討する
- クライアントを扱う第3のドメインが追加され、拡張テーブルの構成（client_id 必須・一意参照）が実態に合わなくなった場合
- 複数利用者・案件ごとのアクセス制限（要件の Won't 項目）を導入する場合。共有クライアントの参照権限の設計が必要になる

## コンテキスト

ADR 管理機能（[要件定義書](../requirements/adr_management_requirements.md)）の導入にあたり、「クライアント（取引先）」という概念が既存の工数管理ドメイン（`WorkHour::Client`、テーブル `work_hour_clients`）と ADR 管理ドメインの双方に必要となった。要件 3.2 は両ドメインのドメインモデル分離を制約として定めている。

### 現状の問題点

- 既存の `WorkHour::Client` は code（一意）・name を自身のテーブルに保持しており、ADR 管理が独自にクライアントを持つと、同一の取引先が2つのドメインに二重登録され、表記揺れ・突き合わせ不能が発生する
- かといって ADR 管理から `work_hour_clients` を直接参照すると、工数管理ドメインが事実上の共有マスタとなり、ドメイン分離の制約に反する

### 制約条件

- ADR 管理と工数管理のドメインモデルは分離する（要件 3.2）
- 取引先という実体はドメインを問わず同一であり、code による同一視ができる必要がある（[用語集](../glossary/adr_management_glossary.md)「共有クライアント」）
- 既存の工数管理機能（Web 管理画面・MCP ツール）は移行後も従来どおり動作すること（要件 7 受け入れ条件）

### 影響範囲（既存コードの `WorkHour::Client` / code・name 参照箇所）

リファクタリングで共有クライアント経由の参照に追随させる必要がある箇所：

- モデル: `app/models/work_hour/client.rb`（code・name の検証、`generate_code_from_name`）、`app/models/work_hour/project.rb`（`belongs_to :client`）
- MCP ツール: `app/mcp/tools/create_work_hour_client_tool.rb`（code 重複チェック・作成）、`find_work_hour_client_tool.rb`（code 完全一致・前方一致、name 部分一致の検索）、`list_work_hour_clients_tool.rb`（code 順の一覧）、`list_work_hour_projects_tool.rb` / `create_work_hour_project_tool.rb`（client_code による特定）
- Web 管理画面: `app/controllers/admin/work_hour/clients_controller.rb`（CRUD、`permit(:code, :name)`）、`app/views/admin/work_hour/clients/`（一覧・詳細・フォーム）、`app/views/admin/work_hour/projects/_form.html.erb`（name によるクライアント選択）
- サービス: `app/services/work_hour/csv_importer.rb`（name によるクライアント検索・`generate_code_from_name` による自動作成）

## 決定

取引先のドメインによらない一般属性を共有テーブルに抽出し、各ドメインはドメイン特化のクライアント拡張から共有テーブルを参照する「共有クライアント基盤＋ドメイン別拡張」アーキテクチャを採用することを決定した。

「ユーザー（取引先）」にすべてを詰め込むのではなく、ドメインごとにドメイン特化の拡張を持たせる発想であり、共有カーネルは識別属性に限定する。

### 実装方針

1. **共有クライアント基盤**: 新設テーブル `clients`（モデル `Client`）に一般属性 code・name を持たせる。code は全ドメイン一意
2. **工数管理拡張（リファクタリング）**: 既存 `work_hour_clients` に共有クライアントへの参照（client_id、必須・一意）を追加し、code・name は `clients` へ完全移行して `work_hour_clients` からは撤去する。上記影響範囲の参照箇所は共有クライアント経由に追随させる
3. **ADR 管理拡張（新設）**: `adr_management_clients` は client_id（必須・一意）のみを持ち、案件（Engagement）の親アンカーと将来の拡張点を担う。ドメイン固有属性は現時点でなし
4. **同一実体の同一視**: ADR 管理のクライアント作成は「共有クライアントを code で検索し、あればそれを参照、なければ作成」とする。工数管理で登録済みの取引先は code 一致で自動的に同一実体になる
5. **データ移行**: 既存 `work_hour_clients` からの一般属性（code・name）の抽出・移行をリリース時作業として実施する（要件 5.1）

## 結果

### ポジティブな影響

1. **同一実体の一元管理**
   - 取引先の二重登録・表記揺れが構造的に発生せず、全ドメイン一意の code で突き合わせが保証される

2. **ドメインの対称性と独立した進化**
   - 工数管理・ADR 管理は対等な拡張としてそれぞれの固有属性・振る舞いを自ドメイン側に持つ。あるドメインの廃止・変更が他ドメインを巻き込まない
   - 共有側の変更理由が識別属性に限定され、共有カーネルの肥大化を防げる

3. **将来ドメインの追加が容易**
   - 新たなドメインは拡張テーブル（client_id 必須・一意参照）を追加するだけで共有クライアントに参加できる

### ネガティブな影響・トレードオフ

1. **共有カーネル結合の意図的な許容**
   - 各ドメインの拡張 → 共有クライアントは、Model 強度の結合が異なる境界づけられたコンテキストをまたぐ構成（機能設計書 2.2 結合評価#2）であり、結合の均衡としては本来避けたい組み合わせである
   - 対策: 共有カーネルの範囲を識別属性（code・name）のみに限定して意図的に許容する。ドメイン固有属性は必ず各拡張側に置き、共有側の変更理由を最小化する（再評価条件参照）

2. **既存 WorkHour 機能のリファクタリング・データ移行コスト**
   - 影響範囲（コンテキスト参照）の参照箇所すべての改修と、既存 `work_hour_clients` データからの一般属性の抽出・移行がリリース時作業として発生する
   - 対策: 移行後の既存の工数管理機能（Web・MCP とも）の回帰確認が要件の受け入れ条件に含まれており、移行の完了条件として担保される

## 代替案

### 案1: ADR管理クライアントが work_hour_clients への任意参照を持つ（当初設計案）

**概要**: ADR 管理のクライアントに `work_hour_client_id`（任意）を持たせ、既存の工数管理クライアントと紐づける。

**却下理由**: 工数管理ドメインが事実上の共有マスタ化し、ドメインの対称性がない。工数管理を廃止・変更すると ADR 管理が巻き込まれ、ドメイン分離の制約（要件 3.2）の趣旨に反する。

### 案2: code 文字列の一致による緩い連携

**概要**: 両ドメインがそれぞれ code・name を保持し、code 文字列の一致で同一実体とみなす。

**却下理由**: 外部キーによる参照整合性がなく、突き合わせが保証されない。片方での code 変更・削除で同一視が静かに壊れる。

### 案3: 完全に独立（連携なし）

**概要**: 両ドメインが独立にクライアントを管理し、連携しない。

**却下理由**: 同一実体の二重登録・表記揺れが発生し、取引先を軸にドメインを横断して参照できない。

### 案4: 単一の Client モデルに全ドメインの属性を集約

**概要**: 1つの `Client` モデルに工数管理・ADR 管理双方の属性・関連を集約する。

**却下理由**: ドメインごとの変更理由が単一モデルに集中し肥大化する。「ユーザーにすべてを詰め込む」構造であり、明示的に避けたい構造としてユーザーが指定した。
