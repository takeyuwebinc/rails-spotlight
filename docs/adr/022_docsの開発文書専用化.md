# 022_docsの開発文書専用化

## ステータス

承認済み

## 日付

2026-07-18

## 信頼度

高 — 公開コンテンツは既にデータベースで管理され MCP ツールで作成・更新されており、リポジトリ内原稿は取り込み元としての役割しか持たない。判断基準は [021_設計書のリポジトリ管理廃止](021_設計書のリポジトリ管理廃止.md) と同じ「二重管理の排除」である。

## 再評価条件

- コンテンツをデータベース外（リポジトリ等）でバージョン管理する必要が生じた場合
- 記事執筆ワークフローをこのリポジトリで再開する場合

## コンテキスト

`docs/` はサイト公開用コンテンツ原稿の管理の場として使われてきた（`guidelines/`・`templates/`・`drafts/`・`published/`）。ADR 021 でもこの用途を主目的として扱い、開発文書のみを整理対象とした。

しかし、公開コンテンツ（プロジェクト・Uses・登壇実績・スライド）はデータベースで管理され、MCP ツールで直接作成・更新できる。リポジトリ内の原稿は `db:import` タスクによる取り込み元に過ぎず、データベースとの二重管理となっている。これは ADR 021 で設計書を廃止したのと同じ構造の問題である。

### 現状の問題点

- **原稿とデータベースの二重管理**: MCP ツールでコンテンツを直接更新すると、リポジトリ内の原稿と乖離する
- **リポジトリの役割の不明確さ**: 本アプリの開発文書とコンテンツ原稿・執筆ガイドラインが同居し、`docs/` の役割が二重になっている

### 制約条件

- 削除した原稿・ガイドラインは Git 履歴に残るため、必要になった場合は履歴から参照できる

## 決定

`docs/` から記事執筆関連（`guidelines/`・`templates/`・`drafts/`・`published/`）をすべて削除し、今後 `docs/` で記事の管理・作成を行わないことを決定した。`docs/` は本アプリについての開発文書（`adr/`・`requirements/`・`reports/`）のみとする。

### 実装方針

1. `docs/guidelines/`・`docs/templates/`・`docs/drafts/`・`docs/published/` を削除する
2. コンテンツの作成・更新は MCP ツール（データベース直接管理）で行う
3. `docs/README.md` を開発文書のみの構成に書き換える
4. 執筆系の `.claude/` 設定（writing・article-review プロファイル、記事関連コマンド、文体ガイド）も削除する。今後このリポジトリで記事執筆は行わない

## 結果

### ポジティブな影響

1. **コンテンツの情報源の一本化**
   - データベースのみが正となり、原稿との乖離が発生しなくなる

2. **`docs/` の役割の明確化**
   - `docs/` は本アプリの開発文書（コードから復元できない情報）のみとなる

### ネガティブな影響・トレードオフ

1. **記事執筆ワークフローの喪失**
   - `.claude/` の執筆系プロファイル・コマンド（writing・article-review・project-draft・uses-item-draft・記事評価レポート）と文体ガイドは `docs/` 配下の原稿・ガイドラインを前提としており、本決定とあわせて削除する
   - 対策: 記事執筆ワークフロー自体を本リポジトリで行わない。必要になった場合は Git 履歴から参照する

2. **`db:import` タスクの既定取り込み元の消滅**
   - `lib/tasks/import.rake` の既定パス `docs/published/` が存在しなくなる
   - 対策: タスクは引数で任意のディレクトリを指定可能なため、外部に原稿がある場合は引き続き使用できる

3. **執筆ガイドライン・原稿の喪失**
   - カスタムMarkdown構文リファレンスを含め Git 履歴を辿らないと参照できなくなる
   - 対策: 必要が生じた時点で履歴から参照する

## 代替案

### 案1: 現状維持（docs でコンテンツ原稿を管理し続ける）

**概要**: `docs/published/` を正の原稿置き場とし、`db:import` で取り込む運用を続ける。

**却下理由**: MCP ツールによるデータベース直接更新と併存する限り二重管理と乖離が避けられない。

### 案2: コンテンツ原稿を別リポジトリへ移管する

**概要**: 原稿・執筆ガイドラインを専用リポジトリに移し、本リポジトリはアプリのみとする。

**却下理由**: データベースが正である以上、場所を移しても原稿との二重管理は解消しない。
