# 025_自然言語検索方式の選定

## ステータス

承認済み

## 日付

2026-07-18（PoC を経て 2026-07-20 承認）

## 信頼度

高 — 比較調査（調査報告書、2026-07-18）で料金・データ取り扱い・索引運用の差を一次情報ベースで確認し、PoC（2026-07-20）で API 仕様（OpenAI 互換・1024次元・512トークン上限・プレフィックス自動付与なし）と日本語検索品質（語彙一致に強く言い換えに弱い特性）を実測で確認済み。判明した弱点（512トークン上限・言い換え）には実装方針で対策を定めた。

## 再評価条件

- 実運用で検索の取り逃がしが頻発した場合（多クエリ検索・キーワード併用でも不十分なら、埋め込み絞り込み＋さくらのチャット API による再ランクのハイブリッド、または索引レス LLM 判定方式へ切り替える）
- ADR 件数の増加により、ローカル全件コサイン類似度計算の検索レイテンシが実用上問題になった場合（sqlite-vec 拡張への移行を検討する）
- さくらのAI Engine の無償枠（埋め込み API 月 10,000 リクエスト）や料金体系が改定された場合
- さくらのAI Engine のデータ取り扱い（国内完結・学習利用なし）の記載・約款が変更された場合

## コンテキスト

ADR 管理機能は、「認証まわりで過去に決めたことは？」のような自然言語の問いで関連 ADR を探せる自然言語検索を Must 要件としている（要件定義書 2.3.1）。要件定義書 3.2 は「自然言語検索の実現方式は要件では定めず、技術選定（ADR）で決定する」と定め、機能設計書も検索インデックスを概念として抽象化し（3.3）、更新トリガーと方針のみ定義して（8章）、方式の確定を本 ADR に委ねている。

### 制約条件

- 環境は Rails 8.1 / SQLite3（SQLite 拡張は未使用）。Solid Queue は gem 導入済みだが未稼働
- ADR には取引先（クライアント）の機密情報が含まれるため、外部サービスへ本文を送る場合のデータ所在・学習利用の有無が重要な評価観点となる
- 利用者は開発者1名と Coding Agent、データ規模は個人受託の範囲で小規模
- 索引が一時的に古くても、キーワード・属性検索と全文参照は DB から正しい結果を返すこと（機能設計書 3.3 の契約。索引は自然言語検索専用）

実現方式の候補比較は[調査報告書](../reports/adr_management_natural_language_search_research.md)に集約されている。本 ADR はその結論の凝縮であり、各方式の API 仕様・料金・出典の詳細は報告書を参照。

## 決定

**さくらのAI Engine の埋め込み（embedding）API（`/v1/embeddings`、multilingual-e5-large）と、SQLite ローカルのベクトル検索（全件コサイン類似度計算）の組み合わせを第一候補として採用することを提案する。**

### 選定理由（調査報告書の結論より）

1. **データ機密性**: 国内のデータセンターでの処理・学習利用なしを公式が明記しており、かつ外部サービスを保存先としない。外部送信は登録・更新時の本文と検索クエリに限られ、ベクトルと本文の保存先はローカル SQLite
2. **コスト**: 埋め込み API は月 10,000 リクエストの無償枠があり、本件の規模（登録・更新時 1 リクエスト＋検索ごと 1 リクエスト）なら 0 円運用が見込める
3. **索引運用の単純さ**: 索引がローカルのベクトル列のみで、外部ドキュメント ID の対応表・削除漏れ対策が不要。機能設計書 3.3・8章の契約（更新依頼・再構築・DB 側の正しさ維持）を最も単純に満たし、誤登録 ADR の削除（要件 2.2.7）も DB 内で完結する
4. **ロックインの小ささ**: モデル・ベンダー切替は全件再埋め込みで完了する
5. **既存環境との適合**: 小規模のため SQLite 拡張なしの全件コサイン類似度計算で成立する。専用拡張は当面導入せず、件数増加時に sqlite-vec（pre-v1 である点に注意）への移行を検討する

### 実装方針

1. ADR の登録・更新時に埋め込み API を呼び、ベクトルを SQLite に保存する。削除時は行削除のみ
2. **チャンク分割**: モデルの入力上限が 512 トークン（日本語で約760文字。PoC で実測）のため、ADR 本文は項目単位（タイトル＋コンテキスト／決定／結果 等）でチャンク分割して埋め込み、検索時は ADR 単位でチャンクの最良スコアを関連度として採用する
3. 自然言語検索時は検索文を埋め込み、ローカルで全件コサイン類似度を計算して関連度として返す（要件 2.3.4 の要約形式）
4. **プレフィックスは実装側で付与する**（`query: ` / `passage: `）。PoC でサーバ側の自動付与がないことを確認済み
5. **言い換え対策**: 埋め込み検索は語彙一致に強く言い換えに弱い（PoC で実測。「デプロイ」直接語彙は1位、「リリース作業」等の言い換えは下位に沈む）。配布 Skill で「複数の言い回しでクエリを投げ結果を統合する」「キーワード・属性検索を併用する」手順を必須化して補う
6. 索引の再構築は全件再埋め込みで行う（機能設計書 8.2 の運用手段）

### PoC 結果（2026-07-20 実施。採用条件はすべて確認済み）

1. `/v1/embeddings` は OpenAI 互換形式（文字列・配列入力、1024次元、usage 返却）。プレフィックスのサーバ側付与は**なし**（raw と付与済みテキストのベクトルが一致しないことで確認）
2. 入力上限は **512 トークン**（超過は HTTP 400。API側での切り詰めはされない）
3. 日本語検索品質: ADR 相当10件×5クエリで top-1 正解率 4/5。外れは語彙の言い換え（「リリース作業」→デプロイ ADR が8位）に起因し、直接語彙なら1位。プレフィックス有無による正解率差は小規模テストでは僅少
4. 詳細な測定値は調査報告書の「PoC 結果」を参照

## 結果

### ポジティブな影響

1. 機密性・コスト・索引運用・ロックインの各観点で本件の条件（個人利用・小規模・機密性重視）に適合する（選定理由のとおり）
2. 外部状態を持たない索引運用により、索引の陳腐化時も DB 側の正しさが保たれ、復旧は全件再埋め込みで完結する

### ネガティブな影響・トレードオフ

1. **環境構築要件の発生**: さくらのAI Engine の契約・API キー等の環境構築要件が発生する（承認に伴い要件定義書 5.2 へ追記済み）
2. **索引更新と外部 API 呼び出しの緊張関係**: 索引更新が外部 API 呼び出しを伴うため、「登録・更新を待たせない／失敗時に本体を失敗させない」方針（機能設計書 8.2）との整合が必要。タイムアウト付きのベストエフォート同期か非同期かは 8.2 具体化時の設計判断となり、**非同期化を選ぶ場合は Solid Queue の稼働化作業（Active Job アダプタ設定・キュー用スキーマ整備）が必要**（gem は導入済みだが未稼働）
3. **プレフィックス・チャンク運用の負担**: サーバ側のプレフィックス自動付与はなく（PoC で確認）、実装側での付与とチャンク分割（512トークン上限対応）が恒常的な実装責務となる
4. **言い換えクエリの弱さ**: 埋め込み検索は言い換えに弱い（PoC で実測）。配布 Skill の多クエリ・キーワード併用手順で運用補完するが、取り逃がしリスクはゼロにならない（頻発時は再評価条件によりハイブリッド再ランクへ）
5. **スケール上の限界**: 全件コサイン類似度計算は件数に線形で、件数増加時は sqlite-vec への移行が必要になる（再評価条件）。sqlite-vec は pre-v1 で破壊的変更がありうる
6. **未確認事項の残存**: さくらのAI Engine の「学習利用なし・国内完結」はサービス紹介ページの記載であり、約款条文・API ログの保持有無は未確認（調査報告書「裏取りできなかった事項」参照）

## 代替案

### 案2: さくらのAI Engine RAG 機能

**概要**: マネージドなベクトルストア＋検索 API（documents_upload / documents_query）。チャンク分割・ベクトル化・格納までがマネージド。

**却下理由**: (1) 無償枠がなく（3円/100チャンク）、課金周期（月額継続か登録時一回か）も一次・二次情報で確認できなかった。(2) ADR 本文そのものが外部サービス上に保存され続け、機密情報を含む ADR の管理として第一候補より不利。誤登録 ADR の削除時に外部ストアの消し忘れがあると機密本文が外部に残留する。(3) ADR の作成・更新・削除のたびに外部ドキュメントのライフサイクル管理（ID 対応表・削除漏れ対策）が必要になる。マネージドなチャンク分割・検索が活きるのは文書が大きく件数が多い場合であり、「1ページ以内の ADR × 小規模件数」の本件ではオーバースペック。

### 案3: OpenAI 等の埋め込み API ＋ ローカルベクトル検索

**概要**: ローカル側は第一候補と同一で、埋め込みプロバイダを OpenAI（text-embedding-3 系）に置き換えたもの。プレフィックス規約が不要で、コストも実質無視できる（text-embedding-3-small の場合 $0.02/1M トークン）。

**却下理由**: 学習利用なしは公式に明記されているが、**既定で ADR 本文が国外（米国）で処理される**。日本データレジデンシーは保存のみ対応かつ承認された顧客向けで、個人利用で適用できるかは未確認。取引先機密を含む ADR 本文の扱いとして本件の機密性の評価観点に合わない。不正利用監視のためのログが最大 30 日保持される点も同様。

### 案4: 索引レス LLM 判定方式（第二候補として温存）

**概要**: 検索時に候補 ADR の要約一覧を LLM（さくらのAI Engine のチャット API）に渡して関連判定・ランク付けさせる。索引を持たず、索引の更新・再構築・整合の問題が構造的に消滅する（調査報告書の方式4）。

**今回第一候補としない理由**: (1) 関連度が LLM の導出値となり、コサイン類似度のような一貫した尺度でない（同一クエリでも揺れうる）。(2) ADR 件数に比例して検索レイテンシとトークン消費が線形に増え、コンテキスト上限が構造的な件数上限になる。(3) 検索のたびに複数 ADR の要約一覧を外部送信するため、1 回あたりの送信範囲が第一候補（クエリ文のみ）より広い。ただし小規模なら成立し索引運用が不要という利点は大きいため、**PoC で第一候補の品質が不十分だった場合の代替として温存する**。第一候補で絞り込み LLM で再ランクするハイブリッドも PoC 段階で検討する（いずれもさくらのAI Engine 内で完結し、データ機密性の水準を保てる）。
