# ADR管理機能 用語集

対象範囲: ADR管理機能（`docs/requirements/adr_management_requirements.md`）のドメイン用語。
種: 同要件定義書「6. 用語定義」セクション。本文全体および既存コード（`app/models/work_hour/`）からも収集した。

## 凡例

| 項目 | 説明 |
|------|------|
| 用語 | 正式な用語名（プロジェクト内で統一して使用する名称） |
| 定義 | 用語の意味を簡潔かつ正確に記述 |
| 別名 | 許容される同義語（正式用語の使用を推奨） |
| 禁止表現 | 混乱を招くため使用を避ける表現（括弧内は理由） |
| 英語名 | 対応する英語表記（モデル名・カラム名・enum 値等、コード上の命名の基礎） |
| コンテキスト | この用語が使用される境界づけられたコンテキスト |
| 関連用語 | 併せて参照すべき用語 |

## 境界づけられたコンテキスト

| コンテキスト | 英語名（名前空間） | 説明 |
|-------------|------------------|------|
| ADR管理コンテキスト | `AdrManagement` | 本機能のドメイン。クライアント・案件・プロジェクトのマスタと ADR 本体を扱う |
| 工数管理コンテキスト | `WorkHour` | 既存の工数管理ドメイン。`WorkHour::Client` / `WorkHour::Project` を持つ。ADR管理コンテキストとはモデルを共有しない（要件 3.2） |
| 共通 | — | コンテキストに依存しない用語（MCP ツール、Coding Agent 等） |

**同名異義語の注意**: 「クライアント」「プロジェクト」は両コンテキストに存在するが別モデルである。さらに工数管理コンテキストでは `WorkHour::Project` のコードコメントで「案件」という語が使われている（`budget_hours` は「案件全体の予算工数」）。つまり工数管理側の「案件」≒ `WorkHour::Project` であり、ADR管理側の「案件」（Engagement、プロジェクトの親）とは階層が異なる。コンテキストを跨ぐ会話・文書では「ADR管理の案件」「工数管理のプロジェクト」のように修飾すること。

## enum 値の定義

### ステータス（`status`）

| 日本語 | 英語名（enum 値） | 定義 |
|--------|------------------|------|
| 提案中 | `proposed` | 決定として提案されたが未承認の状態。登録時に指定可能 |
| 承認済み | `accepted` | 決定として有効な状態。登録時に指定可能（過去の決定の登録を想定）。置換対象に指定できる唯一のステータス |
| 却下 | `rejected` | 提案が採用されなかった状態 |
| 廃止 | `deprecated` | 承認済みだった決定が後継なしに無効化された状態 |
| 置換 | `superseded` | 新しい ADR に置き換えられた状態。置換の一体操作によってのみ遷移する |

**許可されるステータス遷移**（英語名: status transition）: `proposed → accepted`、`proposed → rejected`、`accepted → deprecated`（以上が更新操作で可能）、`accepted → superseded`（置換の一体操作のみ）。これ以外の遷移指定はエラー。

### 信頼度（`confidence`）

| 日本語 | 英語名（enum 値） | 定義 |
|--------|------------------|------|
| 高 | `high` | 決定の確からしさが高い |
| 中 | `medium` | 決定の確からしさが中程度 |
| 低 | `low` | 決定の確からしさが低い。属性検索での洗い出し対象 |

### 点検結果（`result`。点検記録の属性）

| 日本語 | 英語名（enum 値） | 定義 |
|--------|------------------|------|
| 未発火 | `no_trigger` | 点検の結果、再評価条件はまだ満たされていない |
| 発火疑い | `suspected` | 再評価条件が満たされた疑いのある事象を観測した。観測メモが必須。より新しい点検で未発火が記録されると疑いは解消された扱いになる（最新の点検結果が有効） |

## 命名の決定記録

2026-07-18 ユーザー確認済み: 1（`AdrManagement::Adr`）・2（`Engagement`）・3（`superseded` / `Supersession`）・5（`accepted` / `deprecated`）は**推奨案で確定**。4（版履歴のモデル名）は ADR 024 で自前の版テーブルを採用したため **`AdrRevision` で確定**（案A）。6 は参考資料を `reference_links` で確定、更新経路 `origin` は推奨のまま実装時に最終確認。

### 1. ADR モデル名と名前空間

| 案 | 内容 | 評価 |
|----|------|------|
| A（推奨） | 名前空間 `AdrManagement`、モデル `AdrManagement::Adr` | 工数管理の `WorkHour` 名前空間と対称的。トップレベルの既存 `Project` / `SpeakingEngagement` とも衝突しない。`AdrManagement::Adr` にやや冗長感はあるがテーブル名 `adr_management_adrs` で機械的に一貫する |
| B | 名前空間 `AdrManagement`、モデル `AdrManagement::DecisionRecord` | 冗長感は解消するが、ユビキタス言語「ADR」とコード名が乖離する |
| C | 名前空間なし、モデル `Adr` | 最短だが、ドメイン分離（要件 3.2）の意図が名前空間で表現されない |

推奨: **A**。補足: クラス名は `ADR` ではなく `Adr` とする（Rails の acronym inflection 設定を不要にするため）。

### 2. 「案件」の英語名

| 案 | 内容 | 評価 |
|----|------|------|
| A（推奨） | `Engagement`（`AdrManagement::Engagement`） | 「クライアントから受託している継続的な開発対象」という契約的・継続的な関係を最も自然に表す。既存 `SpeakingEngagement`（登壇実績）とは名前空間・意味とも異なるため実害はないが、grep 時に混在する点は認識しておく |
| B | `Account` | CRM 慣行では取引先を指すことが多く、「クライアント」と紛らわしい |
| C | `Product` | Fabble のような開発対象物のニュアンスには合うが、受託関係のニュアンスが落ちる。自社プロダクトと誤解されうる |
| D | `Deal` / `Contract` | 単発契約のニュアンスが強く、「継続的な開発対象」に合わない |

推奨: **A（Engagement）**。

### 3. 「置換」の英語名（ステータス値・関連モデル・操作名）

| 案 | 内容 | 評価 |
|----|------|------|
| A（推奨） | ステータス値 `superseded`、関連モデル `Supersession`、操作 `supersede` | ADR 文化圏の標準表現（"Superseded by ADR-NNN"）。adr-tools 等の既存ツールとも整合し、Coding Agent（LLM）が事前知識で正しく解釈しやすい |
| B | `replaced` / `Replacement` / `replace` | 平易だが ADR の慣用から外れる。`replace` は Ruby の String#replace 等と紛らわしい |
| C | `obsoleted` / `Obsolescence` | 「廃止（deprecated）」との意味的距離が近く混同リスクがある |

推奨: **A（superseded / Supersession）**。関連モデルのカラムは `superseding_adr_id`（新）/ `superseded_adr_id`（旧）を想定。

### 4. 「版履歴」の英語名

| 案 | 内容 | 評価 |
|----|------|------|
| A（推奨） | `AdrRevision`（`AdrManagement::AdrRevision`）、概念名 revision | 「版」の語に一致し、`Version` が持つ gem（paper_trail 等）の既定モデル名との衝突・混同を避けられる |
| B | `AdrVersion` | paper_trail を採用する設計になった場合に自然だが、採用しない場合は誤解を招く。実現方式（自前実装か gem か）は設計事項 |
| C | `AuditLog` | 4.1 で「法令上の記録義務はなし」と明記されており、監査の語は過剰 |

推奨: **A（AdrRevision）**。ただし版履歴の実現方式を gem（paper_trail 等）に委ねる設計判断をした場合は B を再検討する。

### 5. 「承認済み」「廃止」の enum 値

| 項目 | 推奨 | 代替案 | 理由 |
|------|------|--------|------|
| 承認済み | `accepted` | `approved` | ADR 標準テンプレート（Nygard 形式）の語。`approved` はワークフロー承認（人の承認行為）の含意が強い |
| 廃止 | `deprecated` | `obsolete` / `retired` | ADR 文化圏の慣用。`superseded`（後継あり）との対比で「後継なしの無効化」を表す |

### 6. その他の確認事項

- 「更新経路」のカラム名は **`origin` で確定**（2026-07-20 実装時）。代替案は `change_source` / `updated_via` だったが、推奨どおり `origin` を採用した。値の形式は MCP 経由が `oauth:<OAuthアプリケーション名>`、レガシー静的トークン認証が `legacy-token`、Web 管理画面が `admin:<管理者メールアドレス>`。
- 参考資料のカラム名は **`reference_links` で確定**（2026-07-18 機能設計時）。`references` は SQL 予約語だが Rails は識別子を引用するため動作自体は可能。実質的な回避理由は Rails の関連 DSL（`t.references`）・クエリメソッド（`Relation#references`）との紛らわしさ。
