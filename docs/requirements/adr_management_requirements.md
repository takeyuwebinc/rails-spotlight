# ADR管理機能 要件定義書

## 1. 機能概要

受託開発の案件ごとに作成される ADR（アーキテクチャ決定記録）を、本アプリケーションで一元管理する。

現状、ADR は案件ごとの各リポジトリの中で管理されており、次の問題が発生している：

- 1つの案件が複数リポジトリ・複数のプロジェクト（年度等の期間単位）にまたがり、決定の記録が分断され全体像が見えない
- リポジトリ・ブランチ単位で ADR 番号が採番されるため、番号の重複が発生する
- ADR 記録のルールが徹底されず、記録されない決定がある

**主要な提供価値**：

- 案件単位で決定の記録が一元化され、リポジトリ・プロジェクトをまたいで参照できる
- Coding Agent が MCP を通じて過去の決定を検索・参照してから実装し、決定に反する実装や同じ検討のやり直しを防げる
- Coding Agent が MCP を通じて決定を登録・更新でき、配布 Skill のワークフローにより記録ルールが徹底される
- 一元管理による自動採番で番号重複が構造的に発生しなくなる

**利用者**：開発者本人（1名）と、開発者が各案件リポジトリで動かす Coding Agent。案件ごとのアクセス制限は行わない。

## 2. 機能要件

### 2.1 マスタ管理（クライアント・案件・プロジェクト）

3階層の管理単位をマスタとして管理する：

- **クライアント**：取引先。1つのクライアントは複数の案件を持つ。取引先のドメインによらない一般属性（コード・名称）は共有クライアント基盤で管理し、ADR 管理・工数管理の各ドメインはそれぞれの拡張として共有クライアントを参照する（3.2 参照）
- **案件**：クライアントから受託している継続的な開発対象（例: Fabble）。1つの案件は複数のプロジェクトを持つ。**ADR は案件に対して記録する**
- **プロジェクト**：案件内の期間を持つ開発単位（例: Fabble保守開発2026年度、同2027年度）。期間（開始・終了）を持つ

要件：

1. クライアント・案件・プロジェクトの作成・閲覧（一覧・詳細）・編集ができる（Web 管理画面）。誤作成の後始末のため、配下にデータを持たない空のマスタに限り Web 管理画面から削除できる（配下にデータを持つマスタの削除は今回対象外）
2. クライアント・案件・プロジェクトの一覧（list）・単一取得（find）・作成が MCP ツールからもできる。新しい案件・プロジェクトの発生時に、Coding Agent が人間のマスタ登録を待たずに ADR 登録まで完了できるようにするため

### 2.2 ADR 管理

1. **登録**：ADR を案件に対して登録する。どのプロジェクト（期間）で下した決定かを記録できる。登録時のステータスは「提案中」または「承認済み」を指定できる（過去に下した決定の登録を想定）
2. **記録項目**：組織標準の ADR テンプレート（documentation プラグイン adr-documentation スキルのテンプレート）に準拠する
   - タイトル、ステータス、日付、信頼度、再評価条件、コンテキスト（現状の問題点・制約条件）、決定（実装方針）、結果（ポジティブな影響・ネガティブな影響/トレードオフ）、代替案、参考資料
   - タイトルは一覧・検索結果・置換変遷の表示で ADR を一行で識別するために必須とする
   - ステータスは「提案中・承認済み・却下・廃止・置換」の選択式
   - 信頼度は「高・中・低」の選択式（表記揺れを防ぎ、属性検索で信頼度の低い決定を洗い出せるようにするため）
3. **採番**：案件ごとの連番をシステムが自動採番する。同時に登録が行われても番号は重複せず、削除された ADR の番号は再利用しない
4. **置換の一体操作**：新しい ADR の登録時に、置き換え対象として同一案件の「承認済み」の既存 ADR を1件以上指定できる。指定した場合、新 ADR の登録・旧 ADR のステータス「置換」への変更・新旧の置換関係の記録が一体の操作として行われる。旧 ADR の更新漏れによる「矛盾する承認済み ADR の並存」を防ぐため。置換指定時の新 ADR の初期ステータスは「承認済み」とする（旧 ADR が後継なしに「置換」のまま残る状態を防ぐため）。ステータス「置換」への変更はこの一体操作によってのみ行われる（2.2.6 の更新では「置換」を指定できない）
5. **変遷の追跡**：置換関係を新旧の双方向にたどって、ある決定の変遷を参照できる
6. **更新**：ADR の内容を更新できる。ステータスの変更は「提案中→承認済み」「提案中→却下」「承認済み→廃止」の遷移のみ行える。誤った案件への登録を是正するため、所属案件の変更が Web 管理画面から行える（番号は移動先の案件で再採番される。置換関係を持つ ADR は案件を変更できない）
7. **削除**：誤登録された ADR を削除できる（Web 管理画面のみ。誤登録の後始末が目的であり、正当な決定の変遷は削除ではなくステータスで表現する）。置換関係を持つ ADR は削除できない。削除した ADR の版履歴は併せて削除する

### 2.3 検索・参照

検索・参照は MCP ツールとして提供する（Web 管理画面の絞り込みは 2.5.3 に定める範囲）：

1. **自然言語での関連検索**：「認証まわりで過去に決めたことは？」のような自然言語の問いで、関連する ADR を探せる
2. **キーワード・属性検索**：タイトル・本文のキーワード、案件、プロジェクト、ステータス、信頼度、日付による絞り込みができる
3. **案件横断検索**：検索対象を特定の案件に絞ることも、全案件を横断することもできる（例: 「他の案件では認証をどう選定したか」）
4. **返却粒度**：検索結果は要約形式（番号・タイトル・案件・ステータス・信頼度・日付・関連度）で返し、参照は全文を返す。Coding Agent のコンテキスト消費を抑えつつ、関連性判断に必要な情報を提供するため

### 2.4 MCP ツール

Coding Agent 向けに、以下の操作を MCP ツールとして提供する：

- ADR の検索（2.3 の各検索）・参照・登録（置換指定含む）・更新
- クライアント・案件・プロジェクトの一覧・単一取得・作成

### 2.5 Web 管理画面

既存の管理画面に以下を追加する：

1. クライアント・案件・プロジェクトの一覧・詳細・作成・編集
2. ADR の一覧・詳細・作成・編集・削除
3. ADR 一覧の絞り込み（案件・ステータス・キーワード。プロジェクト・信頼度・日付での絞り込みは MCP の属性検索でカバーするため対象外）
4. ADR 詳細での置換変遷・版履歴の参照

### 2.6 配布 Skill

各案件リポジトリに配置して使う配布 Skill（Coding Agent 向けのワークフロー定義）を成果物として作成する：

1. 実装・設計の前に関連 ADR を検索・参照し、決定に反していないか確認する
2. アーキテクチャ上の決定を下した際に、標準テンプレートに沿って ADR を登録する
3. 決定の変更時は置換指定付きの登録を行う
4. 配布 Skill にはバージョンを表記し、古い配布 Skill を使っていることに気づけるようにする（各リポジトリへの配布・更新の手段そのものは今回のスコープ外）

### 2.7 エラー処理

1. 以下はエラーとし、エラーであることと理由を返す：
   - 存在しないクライアント・案件・プロジェクトを指定した登録・検索
   - 置換対象に別案件の ADR を指定した場合
   - 置換対象に「承認済み」以外（提案中・却下・廃止・置換）の ADR を指定した場合
   - 2.2.6 で定めた以外のステータス遷移を指定した場合
2. MCP ツールのエラー応答には、エラー種別・原因となったパラメータ・推奨される次のアクション（例: 案件が存在しない場合はマスタ作成ツールの案内）を含める。Coding Agent が人間の介入なしに次の行動を判断できるようにするため

### 2.8 優先順位（MoSCoW）

| 優先度 | 要件 |
|--------|------|
| Must | 共有クライアント基盤の導入と工数管理クライアントの移行、マスタ管理（Web の作成・閲覧・編集、MCP の一覧・単一取得・作成）、ADR の登録・参照・更新、標準テンプレート項目（タイトル含む）、案件ごとの自動採番、ステータス管理、置換の一体操作と変遷追跡、版履歴（4章）、自然言語検索、キーワード・属性検索、案件横断検索、検索の返却粒度、MCP ツール一式、Web 管理画面（一覧・詳細・作成・編集・絞り込み）、配布 Skill（バージョン表記含む） |
| Should | ADR の削除（Web のみ）、空マスタの削除（Web のみ）、ADR の所属案件の変更（Web のみ） |
| Won't（今回対象外） | 既存 ADR の一括移行機能、配下にデータを持つマスタの削除、配布 Skill の配布・更新手段（Plugin Marketplace 等）、案件ごとのアクセス制限・複数利用者対応、Web 管理画面での自然言語検索 |

## 3. 前提条件・制約事項

### 3.1 前提条件

- Web 管理画面は既存の管理画面認証（ログイン）を、MCP ツールは既存の MCP 向け OAuth 認証基盤を利用する
- 利用者は開発者本人1名とその Coding Agent のみ

### 3.2 制約事項

- ADR 管理のドメインモデルは、既存の工数管理ドメイン（WorkHour::Client / WorkHour::Project）と分離する。ただしクライアントの一般属性（コード・名称）はドメイン横断の共有クライアント基盤として抽出し、各ドメインが自ドメインに特化したクライアント拡張から参照する。これに伴い既存の工数管理クライアントも共有基盤を参照する形へリファクタリングする
- 自然言語検索の実現方式は要件では定めず、技術選定（ADR）で決定する。比較調査（docs/reports/adr_management_natural_language_search_research.md）と PoC を経て、さくらのAI Engine の埋め込み API＋ローカル検索方式を ADR 025 で承認済み（2026-07-20）

## 4. 監査・記録要件

### 4.1 記録の目的

- Coding Agent が ADR を更新するため、意図しない書き換え・誤更新が起きた際に変更内容の確認と復元ができること（業務追跡。法令上の記録義務はなし）

### 4.2 対象操作

- ADR の作成・更新（置換の一体操作による旧 ADR のステータス変更を含む。読み取りは対象外。マスタの変更は対象外）

### 4.3 記録すべき情報

- 変更前の内容（過去の版として参照できること）
- 変更日時
- 更新経路（どの経路・どの OAuth アプリケーション/Agent からの変更か。人間が Web 管理画面から変更した場合はその旨）

### 4.4 保存期間

- 定めなし（版履歴は削除しない。ただし ADR 本体を削除（2.2.7）した場合は、その版履歴も併せて削除する）。法定保存期間なし

### 4.5 参照者

- 開発者本人。誤更新の調査・過去の記述の確認の場面で参照する

### 4.6 PII（個人情報）の取り扱い

- 記録対象に個人情報は含まれない想定。マスキング不要

## 5. リリース・運用要件

### 5.1 リリース時作業

- **既存クライアントデータの移行**：既存の工数管理クライアント（work_hour_clients）から一般属性（コード・名称）を共有クライアント基盤へ抽出・移行する（3.2 の制約に伴う作業。移行後も既存の工数管理機能が従来どおり動作すること）
- 既存リポジトリの ADR の取り込みは行わない（必要になった案件から Coding Agent が MCP ツールで順次登録する）
- 利用者は本人のみのため、通知・段階的ロールアウトは不要

### 5.2 環境構築要件

自然言語検索の技術選定（ADR 025 承認済み）に伴う要件：

- **外部サービス**: さくらのAI Engine（埋め込み API）。アカウント・API トークンは取得済み
- **シークレット**: API トークンを Rails credentials（`sakura.ai_account_token`）で管理する。設定済み。対象環境: 本番・開発（テスト・CI は外部 API を呼ばずスタブする）
- 新規インフラリソース・CI/CD 変更・監視追加は不要（索引はアプリの SQLite 内で完結）

## 6. 用語定義

| 用語 | 説明 |
|------|------|
| ADR | アーキテクチャ決定記録（Architecture Decision Record）。技術的決定の内容・経緯・理由を記録する文書 |
| 共有クライアント | 取引先のドメインによらない一般属性（コード・名称）を持つドメイン横断の共有基盤。各ドメインのクライアント拡張から参照される |
| クライアント | ADR 管理ドメインにおける取引先の拡張。共有クライアントを参照し、案件の親としての役割を担う（工数管理ドメインの WorkHour::Client とは別モデル） |
| 案件 | クライアントから受託している継続的な開発対象（例: Fabble）。ADR の記録先となる単位 |
| プロジェクト | 案件内の期間を持つ開発単位（例: Fabble保守開発2026年度）。工数管理ドメインの WorkHour::Project とは別モデル |
| 置換 | 新しい ADR が既存の ADR の決定を置き換えること。ステータスの一種でもある |
| 変遷 | 置換関係の連なり。ある決定がどの決定に置き換えられてきたかの履歴 |
| 版履歴 | ADR の作成・更新のたびに記録される、変更前内容・変更日時・更新経路の記録 |
| 配布 Skill | 各案件リポジトリに配置し、Coding Agent に ADR の検索・参照・登録のワークフローを徹底させる指示書 |
| Coding Agent | 開発者が各案件リポジトリで使用する AI コーディングエージェント（Claude Code 等） |
| 一覧（list）/単一取得（find） | MCP ツールの読み取り操作の区分。一覧は条件に合う複数件、単一取得は識別子指定による1件の取得 |

## 7. 受け入れ条件

- [ ] クライアント・案件・プロジェクトを Web 管理画面と MCP ツールの双方から作成・参照できること
- [ ] 共有クライアント基盤への移行後、既存の工数管理機能（Web・MCP とも）が従来どおり動作すること
- [ ] ADR を案件に対して登録でき、案件ごとの連番が自動採番されること（同時登録でも重複しないこと）
- [ ] 置換指定付きの登録で、新 ADR の登録・旧 ADR のステータス変更・置換関係の記録が一体で行われること
- [ ] 置換変遷を新旧の双方向にたどって参照できること
- [ ] 一体操作以外の手段でステータスを「置換」に変更できないこと
- [ ] 自然言語・キーワード/属性・案件横断の各検索で該当する ADR が見つかり、検索結果が要約形式（2.3.4 の項目）で返ること
- [ ] ADR の作成・更新のたびに版履歴（変更前内容・日時・更新経路）が記録され、参照できること
- [ ] Web 管理画面の ADR 一覧で案件・ステータス・キーワードによる絞り込みができること
- [ ] 配布 Skill が作成され、バージョンが表記されていること
- [ ] ADR の所属案件の変更（Web のみ）で移動先での再採番と版履歴の記録が行われること。置換関係を持つ ADR では変更できないこと
- [ ] 配下にデータを持たない空のマスタのみ削除でき、配下にデータを持つマスタの削除は拒否されること
- [ ] 未認証のアクセス（Web・MCP とも）が拒否されること
- [ ] 2.7.1 の各エラーケース（存在しないマスタの指定、別案件・不正ステータスの置換対象指定、不正なステータス遷移）でエラー種別・原因・次のアクションを含む応答が返ること
