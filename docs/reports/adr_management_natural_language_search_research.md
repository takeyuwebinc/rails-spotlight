# ADR管理機能 自然言語検索の実現方式 比較調査報告書

**ステータス**: Completed

## 概要

### 調査の背景

ADR管理機能（要件定義書 2.3.1）は、「認証まわりで過去に決めたことは？」のような自然言語の問いで関連 ADR を探せる「自然言語検索」を Must 要件としている。要件定義書 3.2 は「自然言語検索の実現方式は要件では定めず、技術選定（ADR）で決定する」と定めており、機能設計書も検索インデックスを「概念」として抽象化した上で、実現方式の確定を技術選定 ADR（ADR 025）に委ねている（機能設計書 2.1・3.3・8章）。

本報告書は、その ADR 025（技術選定）の判断材料として、実現方式の候補を比較調査した結果をまとめる。

### 調査の目的

以下を明らかにする：

1. 候補となる各実現方式の API 仕様・料金・データ取り扱い（一次情報ベース）
2. 評価軸（検索品質・索引運用・コスト・データ機密性・実装コスト・依存/ロックイン・既存環境適合）ごとの比較
3. 推奨案とその採用条件

### 調査範囲

- **対象**: 自然言語検索の実現方式のみ。キーワード・属性検索は DB への問い合わせで実現する方針（機能設計書 2.1）のため対象外
- **前提環境**: Rails 8.1 / SQLite3（FTS5 等の拡張は未使用）/ Solid Queue は gem 導入済みだが未稼働（後述の自己検証参照）/ 利用者は開発者1名と Coding Agent / データ規模は個人受託の範囲で小規模
- **重要評価観点**: ADR には取引先（クライアント）の機密情報が含まれるため、外部サービスへ本文を送る場合のデータ所在・学習利用の有無を必ず評価する
- **対象外**: 検索品質の実測ベンチマーク（本調査は文献調査。実測は PoC として次のアクションに含める）、Web 管理画面での自然言語検索（要件定義書で Won't）

## 調査内容

### 調査対象

| # | 方式 | 概要 |
|---|------|------|
| 1 | さくらのAI Engine RAG 機能 | マネージドなベクトルストア＋検索 API（documents_upload / documents_query / documents_chat） |
| 2 | さくらのAI Engine 埋め込み API ＋ ローカルベクトル検索 | `/v1/embeddings` でベクトル化し、SQLite ローカルで類似度計算（全件コサイン類似度、または sqlite-vec 拡張） |

比較対象として Anthropic の埋め込み API も検討したが、Anthropic は自社の埋め込みモデルを提供しておらず、公式ドキュメントは外部プロバイダ（Voyage AI）を紹介している（[Anthropic 公式ドキュメント](https://platform.claude.com/docs/en/build-with-claude/embeddings)）。このため方式3の代表は OpenAI とした。

## 調査結果

### 前提となる既存環境の確認（自己検証）

リポジトリを確認した結果（2026-07-18 時点、自己検証）：

- `Gemfile` に `gem "sqlite3", ">= 2.1"`（ロック済みバージョンは 2.6.0）と `gem "solid_queue"` がある
- Active Job のアダプタ設定（`queue_adapter`）・`config/queue.yml`・キュー用スキーマ（`db/queue_migrate`）は存在せず、**Solid Queue は未稼働**。ただし production の `database.yml` には queue 用データベース定義が既に存在する
- `database.yml` に `extensions:` 設定はなく、SQLite 拡張は未使用

Rails 8.1 以降は `config/database.yml` の `extensions:` キーで SQLite 拡張（sqlite-vec 等）をロードできる（sqlite3 gem v2.4.0 以降が必要。本リポジトリは Rails 8.1・sqlite3 2.6.0 で条件を満たす）（[Rails PR #53827](https://github.com/rails/rails/pull/53827)、[This Week in Rails 2024-12-06](https://rubyonrails.org/2024/12/6/this-week-in-rails)）。

### 方式1: さくらのAI Engine RAG 機能

**API 仕様**（[操作ガイド（公式マニュアル）](https://manual.sakura.ad.jp/cloud/ai-engine/03-operation-guide.html)）：

- `documents_upload`（`https://api.ai.sakura.ad.jp/v1/documents/upload/`）でドキュメントをアップロードすると、チャンク分割・ベクトル化・ベクトルストアへの格納までがマネージドで行われる
- `documents_query` はベクトルストアへの自然言語検索（チャンク単位で結果を返す）、`documents_chat` は検索結果を使った LLM 回答生成
- 埋め込みモデルは `multilingual-e5-large`（デフォルト）と `preview/Qwen3-Embedding-4B-FP16` を `model` パラメータで指定可能
- 距離関数は `distance_type` パラメータでコサイン距離（`cosine`）と L2 距離（`l2`）を選択可能。チャンクサイズは `chunk_size` パラメータで指定可能（2026-02-25 の機能追加。[さくらのクラウドニュース](https://cloud.sakura.ad.jp/news/2026/02/25/ai-engine_rag-api/)）
- クエリ時に `tags` パラメータで検索対象ドキュメントを絞り込める（[さくらのAI Engine ことはじめ（さくら公式コラム）](https://ai.sakura.ad.jp/column/ai-engine-getting-started-chat-rag-mcp/)）。案件別の絞り込み・案件横断検索はタグ設計で実現できる見込み
- チャンクは既定で 512 トークン単位に分割され、ベクトルと元テキストがペアで保存される（同上）
- 検索応答には類似度（distance）とチャンク本文が含まれる（[NETASSIST 技術ブログ（二次情報・実行例）](https://www.netassist.ne.jp/sakuracloud-labo/archives/809)）。要件 2.3.4 の「関連度」への変換は可能と見られるが、`documents_query` の正式なレスポンススキーマは一次情報で未確認

**料金**（[公式サービスページ](https://ai.sakura.ad.jp/sakura-ai/ai-engine/)）：

- RAG（ドキュメント）は**無償利用枠なし**。3円/100チャンク（100チャンク単位）で、基盤モデル無償プラン利用中でも課金対象
- 課金周期について、公式ページの表記からは「保存中ずっと月額」か「登録時一回」かを確認できなかった。二次情報（[NETASSIST 技術ブログ](https://www.netassist.ne.jp/sakuracloud-labo/archives/809)）も「アップロードした分だけ費用が発生し、不要なドキュメントは削除すべき」という趣旨の記載にとどまり、課金周期は明示されていない。**契約前に要確認**
- 参考: 既定チャンク 512 トークンの場合、100チャンク ≒ 51,200 トークン分の本文に相当する。個人受託規模の ADR 全文がこの範囲に収まり、かつ仮に月額継続課金だとすれば月額数円〜数十円のオーダーと推測されるが、これは課金周期・件数・本文量に依存する推測である

**データ取り扱い**：

- 公式サービスページに「フルマネージド環境で、データは国内で完結。モデル提供元にも送られないため、学習に利用される心配もありません」と明記（[公式サービスページ](https://ai.sakura.ad.jp/sakura-ai/ai-engine/)）
- さくらインターネットが運営する国内のデータセンターでの処理（[さくらインターネット ニュースリリース](https://www.sakura.ad.jp/corporate/information/newsreleases/2025/09/24/1968221046/)）
- 方式1では **ADR 本文そのものが外部サービス上に保存され続ける**（ベクトルと元テキストのペアが保存されるため）。学習利用なし・国内完結ではあるが、外部保存という点で方式2〜4と異なる
- 上記はサービス紹介ページの記載であり、約款・利用規約の条文レベルでは未確認

**索引の運用**：

- ADR の作成・更新・削除のたびにドキュメントの登録・削除（更新はドキュメント管理 API での名前・タグ変更を除き、削除→再アップロードになる見込み。内容更新 API の有無は一次情報で未確認）
- ADR と外部ドキュメント ID の対応表をアプリ側で管理する必要がある
- 索引再構築は「全ドキュメント削除→全件再アップロード」

### 方式2: さくらのAI Engine 埋め込み API ＋ SQLite ローカルベクトル検索

**API 仕様**：

- `https://api.ai.sakura.ad.jp/v1/embeddings` が提供され、チャット API は OpenAI 互換形式（`/v1/chat/completions`）・Anthropic 互換形式（`/v1/messages`）が公式マニュアルに明記されている（[利用手順（公式マニュアル）](https://manual.sakura.ad.jp/cloud/ai-engine/02-howto.html)）。embeddings エンドポイントの詳細仕様（リクエスト形式）は同ページに記載がなく未確認
- 埋め込みモデルは `multilingual-e5-large`（および preview の Qwen3-Embedding-4B）（[公式サービスページ](https://ai.sakura.ad.jp/sakura-ai/ai-engine/)）
- **注意（検索品質に直結）**: multilingual-e5 系モデルは、入力テキストへの `query: ` / `passage: ` プレフィックス付与を前提に学習されており、省略すると性能が低下すると公式モデルカードに明記されている（[intfloat/multilingual-e5-large（Hugging Face）](https://huggingface.co/intfloat/multilingual-e5-large)）。さくらの embeddings API がこれをサーバ側で付与するかは未確認のため、実装側での付与を前提に PoC で検証する必要がある。同モデルは 100 言語対応（xlm-roberta ベース）で、多言語検索ベンチマーク（MIRACL 等）の学習データを含む

**ローカル側の実現**：

- 小規模データなら、埋め込みベクトルを ADR テーブル（または付随テーブル）に BLOB/JSON で保存し、検索時に全件コサイン類似度を Ruby で計算するだけで成立する。SQLite 拡張・追加ミドルウェアは不要
- 将来件数が増えた場合は sqlite-vec 拡張（`vec0` 仮想テーブルによる KNN 検索）に移行できる。Ruby 向け gem（`sqlite-vec`）があり、`SqliteVec.load(db)` でロードする（[sqlite-vec Ruby ドキュメント](https://alexgarcia.xyz/sqlite-vec/ruby.html)）。Rails 8 の `database.yml` `extensions:` 設定でもロード可能（前掲 Rails PR #53827）
- ただし sqlite-vec は **pre-v1（2026-03-31 時点 v0.1.9）で、破壊的変更がありうると公式 README に明記**されている（[GitHub asg017/sqlite-vec](https://github.com/asg017/sqlite-vec)）

**料金**（[公式サービスページ](https://ai.sakura.ad.jp/sakura-ai/ai-engine/)）：

- 埋め込み（multilingual-e5-large）: **月 10,000 リクエストの無償枠**。超過分は入力 2円/10,000 トークン・出力無料（Qwen3-Embedding-4B は入力 3円/10,000 トークン）
- 消費は「ADR の登録・更新時に 1 リクエスト＋検索クエリごとに 1 リクエスト」程度。月間合計 10,000 リクエスト以内なら**無償枠内（0円）**で運用できる

**データ取り扱い**：

- 方式1と同じ国内完結・学習利用なしの記載が適用される（同一サービス）
- 外部に送るのは**埋め込み計算時の ADR 本文（および検索クエリ）のみ**で、ベクトルと本文の保存先はローカル SQLite。方式1と異なり外部サービス側を保存先としない（外部サービス側で API ログ等として保持されるかどうかは約款未確認）

**索引の運用**：

- ADR の作成・更新時に埋め込み API を 1 回呼んでベクトル列を更新、削除時は行削除のみ。対応表・外部状態が不要で、機能設計書 3.3 の契約（索引が一時的に古くても、キーワード・属性検索と全文参照は DB から正しい結果を返す）との整合が取りやすい
- 索引再構築＝全件再埋め込み（小規模なら数リクエストで完了）。モデル切替時も同じ手順
- 外部 API 呼び出しを伴うため、機能設計書 8.2 の「登録・更新を待たせない／失敗時に本体を失敗させない」方針との整合が必要。埋め込み 1 回の軽量な呼び出しだが、同期呼び出しは「待たせない」条件と緊張関係にある。タイムアウト付きのベストエフォート同期を許容するか、Solid Queue を稼働化して非同期にするかは 8.2 具体化時の設計判断となる

### 方式3: OpenAI 埋め込み API ＋ ローカルベクトル検索

ローカル側は方式2と同一。埋め込みプロバイダのみ異なる。

**料金**：

- text-embedding-3-small: **$0.02 / 1M トークン**（[OpenAI 公式モデルページ](https://developers.openai.com/api/docs/models/text-embedding-3-small)）。無償枠はないが、個人受託規模の全 ADR を埋め込んでも 1M トークンに達しない限り $0.02 未満であり、コストは実質無視できる
- 次元数は既定 1536（`dimensions` パラメータで削減可能）、最大入力 8,192 トークン（[OpenAI 公式 Embeddings ガイド](https://developers.openai.com/api/docs/guides/embeddings)）

**データ取り扱い**（[OpenAI 公式: Your data](https://developers.openai.com/api/docs/guides/your-data)）：

- 「2023-03-01 以降、API に送信されたデータは、明示的にオプトインしない限りモデルの学習に使用されない」と公式に明記
- 不正利用監視のためのログは最大 30 日保持（ゼロデータ保持 ZDR は事前承認制）
- **データレジデンシー**: 日本リージョンは「保存のみ」対応で、処理（推論）を含む対応は米国・EU・UAE 等に限られる。さらにレジデンシー指定は承認された顧客向けであり、個人利用で適用できるかは未確認。**既定では ADR 本文が国外（米国）で処理される**

**プレフィックス**: text-embedding-3 系は e5 系のようなプレフィックス規約を持たず、素のテキストを埋め込める。

**参考（Anthropic）**: Anthropic は埋め込みモデルを提供しておらず、公式ドキュメントは Voyage AI を紹介している（[Anthropic 公式ドキュメント](https://platform.claude.com/docs/en/build-with-claude/embeddings)）。Voyage AI も米国のサービスであり、データ所在の観点は OpenAI と同様に国外処理となる。

### 方式4: 索引レス（検索時に LLM へ候補一覧を渡して関連判定）

**仕組み**：

- 検索時に、DB から候補 ADR（属性で絞り込んだ上での全件）の要約一覧（番号・タイトル・コンテキスト冒頭等）を取り出し、チャット API（LLM）に検索文と併せて渡して、関連する ADR の判定とランク付けをさせる。索引を一切持たない
- 機能設計書 8 章の「検索インデックス更新」責務は no-op になり、索引の更新・再構築・整合の問題が構造的に消滅する。索引の陳腐化が起こらず、常に最新の DB 内容で検索される
- 関連度は LLM に数値出力させる導出値となり、埋め込みのコサイン類似度のような一貫した尺度ではない（同一クエリでも揺れうる）

**料金**（さくらのAI Engine のチャット API を使う場合。[公式サービスページ](https://ai.sakura.ad.jp/sakura-ai/ai-engine/)）：

- Chat Completions は**月 3,000 リクエストの無償枠**。超過分は例: gpt-oss-120b で入力 0.15円・出力 0.75円/10,000 トークン
- 検索 1 回につき 1 リクエスト消費。トークン消費は候補 ADR 件数に比例して増える

**データ取り扱い**：

- さくらのAI Engine を使う限り方式1・2と同じ（国内完結・学習利用なしの記載）。ただし**検索のたびに複数 ADR の要約一覧を送信する**ため、1 回あたりの外部送信範囲は方式2（クエリ文のみ）より広い

**制約・リスク**：

- ADR 件数の増加に比例して、検索レイテンシとトークン消費が線形に増える。コンテキスト上限が実質的な件数上限になる（小規模前提なら当面問題にならないが、スケール上限が構造的に存在する）
- 出力が非決定的で、要約一覧のプロンプト設計と応答パース（構造化出力）の作り込みが必要
- 検索品質は「候補一覧に含めた要約の情報量」に依存する。タイトルのみでは「認証まわりで過去に決めたことは？」のような問いに対する再現率が下がる可能性がある（推測。PoC で確認すべき事項）

## 分析・考察

### 主要な発見

1. **さくらのAI Engine は「RAG 機能」と「埋め込み API」で料金構造が大きく異なる**。RAG は無償枠なしの従量課金（3円/100チャンク。依頼時の事前情報と一致することを一次情報で確認）だが、埋め込み API には月 10,000 リクエストの無償枠があり、本件の規模なら 0 円で運用できる可能性が高い。一方、RAG の課金周期（月額継続か登録時一回か）は一次・二次情報とも確認できなかった
2. **データ機密性の観点では、さくらのAI Engine 系（方式1・2・4）が優位**。国内のデータセンターでの処理・学習利用なしを公式が明記している。OpenAI も学習利用なしは明記しているが、処理は既定で国外（日本リージョンは保存のみ・承認制）
3. **方式1は ADR 本文が外部に保存され続け、方式2はベクトル計算時に送信されるのみ**という差がある。機密情報を含む ADR の管理では、この「外部保存の有無」は約款確認前の重要な差分
4. **既存環境（SQLite・Rails 8.1）はローカルベクトル検索と親和性が高い**。小規模なら拡張なしの全件コサイン類似度で成立し、必要になれば sqlite-vec（pre-v1 である点に注意）へ段階的に移行できる。索引更新を同期で行うか Solid Queue を稼働化して非同期にするかは、機能設計書 8.2 の「登録・更新を待たせない」方針の解釈を含む設計判断として残る（方式4は索引を持たないためこの論点自体がない）
5. **埋め込みモデル multilingual-e5-large にはプレフィックス規約があり、検索品質に直結する**。方式2の実装では必ず考慮が必要で、方式1（RAG）が内部で対応しているかは未確認

### リスクと制約

- **方式1**: 課金周期・ドキュメント内容更新 API・`documents_query` レスポンス仕様が一次情報で未確認。誤登録 ADR の削除（要件 2.2.7）時に外部ストアの消し忘れが起こると、機密本文が外部に残留する
- **方式2**: e5 のプレフィックス運用を誤ると品質劣化（公式モデルカード明記）。sqlite-vec を使う場合は pre-v1 の破壊的変更リスク（当面は全件コサインで回避可能）
- **方式3**: 機密性要件（取引先情報の国外送信回避）を重視する場合、既定構成では採用しづらい。学習利用なしでも 30 日の不正利用監視ログ保持がある
- **方式4**: スケール上限が構造的に存在し、関連度の一貫性がない。LLM 生成を伴うため検索レイテンシは埋め込み方式より大きいと見込まれる（実測なし）
- **全方式共通**: 日本語検索品質は文献のみで実測しておらず、品質の優劣は本調査では確定できない。さくらのAI Engine の「学習利用なし・国内完結」はサービス紹介ページの記載であり、約款条文は未確認

## 結論・推奨事項

### 結論

個人利用・小規模データ・機密性重視という本件の条件では、各方式の主な差は「検索品質」ではなく「データの外部保存の有無」「索引運用の複雑さ」「コスト構造」に現れる。さくらのAI Engine 系はデータ機密性の公式記載（国内完結・学習利用なし）があり、OpenAI は既定で国外処理となる。RAG 機能は無償枠がなく外部に本文が保存され続けるのに対し、埋め込み API は無償枠内で運用でき、データ保存をローカルに保てる。

### 推奨事項

1. **方式2（さくらのAI Engine 埋め込み API ＋ SQLite ローカル全件コサイン類似度）を第一候補として推奨する**
   - 理由: (a) 国内処理・学習利用なしの公式記載があり、かつ外部サービスを保存先としない（外部送信は登録・更新時の本文と検索クエリに限られる。なお「1 回の検索あたりの送信量」では要約一覧を送る方式4より狭い、という関係にある）、(b) 月 10,000 リクエストの無償枠内で 0 円運用が見込める、(c) 索引がローカル列のみで、機能設計書 3.3・8章の契約（更新依頼・再構築・DB 側の正しさ維持）を最も単純に満たす、(d) ロックインが小さく、モデル・ベンダー切替は全件再埋め込みで完了する、(e) SQLite 拡張が当面不要で、索引更新の同期/非同期（Solid Queue 稼働化の要否）を設計時に選択できる
   - 期待効果: 外部状態を持たない索引運用により、誤登録 ADR の削除（要件 2.2.7）やデータ復旧が DB 内で完結する
   - **採用条件**: PoC で (i) `/v1/embeddings` のリクエスト仕様と `query: `/`passage: ` プレフィックスの扱いを確認し、(ii) 実際の ADR 相当の日本語テキストで検索品質を確認すること。品質が不十分な場合は次項へ
2. **品質不足時の代替・補完として方式4（索引レス LLM 判定）を第二候補とする**。方式2で候補を絞り込み LLM で再ランクするハイブリッドも選択肢になる（ハイブリッドは本調査では個別に評価しておらず、PoC 段階で検討する。いずれもさくらのAI Engine 内で完結しデータ機密性の水準を保てる）
   - 理由: 小規模データなら成立し、索引運用が不要という運用上の利点が大きい。曖昧な自然言語の問いへの適合も期待できる（実測は未実施）
3. **方式1（RAG 機能）は本件では推奨しない**
   - 理由: 無償枠がなく（課金周期も未確認）、機密本文が外部に保存され続け、ADR 更新・削除のたびに外部ドキュメントのライフサイクル管理（対応表・削除漏れ対策）が必要になる。マネージドなチャンク分割・検索が活きるのは文書が大きく件数が多い場合であり、「1ページ以内の ADR × 小規模件数」という本件ではオーバースペック
4. **方式3（OpenAI）は機密性要件を緩和できる場合のみ検討対象とする**
   - 理由: 学習利用なしは明記されているが、既定で国外処理となり、取引先機密を含む ADR 本文の扱いとして本件の評価観点に合わない

## PoC 結果（2026-07-20 実施）

`/v1/embeddings` に対して Rails runner スクリプトから実測した（採用条件 (i)(ii) の検証。裏取りできなかった事項 5・8 の一部を解消）。

### API 仕様（採用条件 (i)）

- **OpenAI 互換形式で動作**: `{model, input}` の POST。`input` は文字列・配列（バッチ）とも受理。レスポンスは `id / object / created / model / data / usage`、次元数 **1024**、`usage` にトークン数が返る
- **プレフィックスのサーバ側付与はなし**: 同一テキストの raw と `query: `/`passage: ` 付きのベクトルのコサイン類似度が 0.94〜0.98（≠1.0）であり、入力はそのまま埋め込まれる。**実装側での付与が必要**
- **入力上限は 512 トークン**（モデルのコンテキスト長）。超過は HTTP 400 でエラーメッセージに上限と要求トークン数が明示される。API 側での切り詰めは行われない。日本語は実測で約 0.67 トークン/文字（≒760 文字で上限）。**1ページ相当の ADR 全文は 1 回で埋め込めないため、項目単位のチャンク分割が必要**

### 日本語検索品質（採用条件 (ii)）

ADR 相当の日本語ミニコーパス 10 件 × 自然言語クエリ 5 件で top-1 正解率を測定：

- プレフィックスあり: **4/5**、プレフィックスなし: **4/5**（この規模では正解率への影響は僅少。スコア帯は 0.75〜0.83 に密集しマージンは 0.01〜0.03 と小さい）
- 外れた 1 件の原因分析（デプロイ方式 ADR に対するクエリ）:
  - 「**デプロイ**方法について決めたことは？」（語彙一致）→ **1位**
  - 「**リリース作業**のやり方について決まっていることは？」（言い換え）→ **8位**/10件
  - 「**本番反映**の手順に関する決定は？」（言い換え）→ **10位**/10件
- **結論: 語彙一致に強く、言い換え・類義語に弱い**。単一クエリ・上位N件のみの運用では取り逃がしが発生する

### 判定と対策（ADR 025 に反映済み）

方式2を対策付きで採用（ADR 025 承認済み・2026-07-20）：

1. 項目単位のチャンク分割埋め込み（512 トークン上限対応。ADR 単位でチャンク最良スコアを採用）
2. プレフィックスの実装側付与
3. 言い換え対策として、配布 Skill で「複数の言い回しでの検索と結果統合」「キーワード・属性検索の併用」を必須手順化
4. 取り逃がしが頻発する場合はハイブリッド再ランク（埋め込み絞り込み＋さくらのチャット API）へ移行（ADR 025 再評価条件）
