## 1. 機能概要

### 1.1 目的

サービス案内ページを新設し、想定閲覧者（エンジニアリングマネージャー）がサービス内容を理解し、問い合わせに至るまでの導線を整備する。

- サービス一覧ページと各サービス詳細ページを新規作成する
- ナビゲーションを想定閲覧者の意思決定プロセスに沿った順序に再構成する
- 各ページからGoogle Formsの問い合わせフォームへの導線を設置する

### 1.2 主要機能

1. **サービス一覧表示**: 提供中のサービス（受託開発・技術顧問）を一覧で表示する
2. **サービス詳細表示**: 各サービスの内容を詳細に紹介するページを表示する
3. **問い合わせ導線（CTA）**: 各サービスページからGoogle Formsへのリンクを提供する
4. **ナビゲーション再構成**: ヘッダー・フッター・モバイルメニューの項目と順序を変更する

### 1.3 処理フロー概要

本機能はすべて静的なページ表示であり、データベースアクセスや動的処理は不要。

#### 1.3.1 閲覧者の想定導線

1. ナビゲーションから「Services」をクリックしてサービス一覧ページに到達
2. サービス一覧から個別サービスの詳細ページへ遷移
3. サービス詳細ページのCTAからGoogle Formsの問い合わせフォームへ遷移（外部リンク）

## 2. 画面設計

### 2.1 サービス一覧ページ（`/services`）

#### 2.1.1 画面構成

- **ページヘッダー**: ページタイトルと概要文
  - 既存ページ（Projects、Speaking等）と同じレイアウトパターンに準拠する
- **サービスカード一覧**: 提供中のサービスを並べて表示する
  - 受託開発（Outsourcing）
  - 技術顧問（Technical Advisor）
  - 各カードにサービス名、概要文、詳細ページへのリンクを含む
- **CTA**: Google Formsへのリンクボタン

#### 2.1.2 SEO対応

- ページタイトル、meta description、canonical URLを設定する
- 構造化データ（Service型）を設置する

### 2.2 受託開発 詳細ページ（`/services/outsourcing`）

#### 2.2.1 画面構成

- **ページヘッダー**: サービス名と概要文
- **サービス説明**: 旧サイトの内容をベースに、一人会社の体制に合わせてリライトした説明文
  - 旧サイトでの掲載内容（リライト対象）：
    - 新規Webサービス開発
    - 既存サービスの保守・レガシーシステム対応
    - 契約形態（準委任契約ベースの月額稼働 / 請負契約）
- **CTA**: Google Formsへのリンクボタン
- **サービス一覧への戻りリンク**

### 2.3 技術顧問 詳細ページ（`/services/technical_advisor`）

#### 2.3.1 画面構成

- **ページヘッダー**: サービス名と概要文
- **サービス説明**: 旧サイトの内容をベースに、一人会社の体制に合わせてリライトした説明文
  - 旧サイトでの掲載内容（リライト対象）：
    - 月額契約での技術支援
    - Ruby on Railsの専門知識によるサポート
    - 技術的負債の予防
    - オンライン・対面での相談対応
- **CTA**: Google Formsへのリンクボタン
- **サービス一覧への戻りリンク**

### 2.4 問い合わせCTAの仕様

全サービスページ共通で以下の仕様とする。

- 遷移先: `https://forms.gle/scwNEGrT196rFnD9A`（既存のGoogle Forms）
- 新規タブで開く（外部リンク）
- ボタンのスタイル: 既存のホームページ・AboutページのCTAと同じスタイル（teal系のボタン）

### 2.5 ナビゲーションの再構成

#### 2.5.1 変更対象

ナビゲーションは以下の3箇所に存在し、すべてを変更する。

| 箇所 | 現在の構成 |
|------|-----------|
| デスクトップヘッダー | About / Articles(外部) / Projects / Speaking / Uses |
| モバイルメニュー | About / Articles(外部) / Projects / Speaking / Uses |
| フッター | About / Articles(外部) / Projects / Speaking / Uses |

#### 2.5.2 変更後の構成

| 箇所 | 変更後の構成 |
|------|-------------|
| デスクトップヘッダー | Services / Projects / Speaking / About / Uses |
| モバイルメニュー | Services / Projects / Speaking / About / Uses |
| フッター | Services / Projects / Speaking / About / Uses |

#### 2.5.3 変更のポイント

- **Servicesを先頭に配置**: 想定閲覧者（エンジニアリングマネージャー）がまず知りたい「何を提供するか」を最初に提示
- **Projects / Speakingを中盤に**: 実績・信頼性を示す情報を次に配置
- **About / Usesを後方に**: 人物・補足情報は関心を持った後に参照する想定
- **Articlesをナビゲーションから除外**: 外部リンク（Zenn）のため、Aboutページ内にリンクを残す

### 2.6 Aboutページの変更

- Articlesをナビゲーションから除外するため、Aboutページ内にZennへのリンクを追加する
- 既存のソーシャルリンク（X、GitHub）と同じ並びに配置する

## 3. ルーティング

### 3.1 新規ルート

| パス | 動作 |
|------|------|
| `/services` | サービス一覧ページの表示 |
| `/services/outsourcing` | 受託開発詳細ページの表示 |
| `/services/technical_advisor` | 技術顧問詳細ページの表示 |

### 3.2 既存ルートの変更

| パス | 変更前 | 変更後 |
|------|--------|--------|
| `/services` | `/projects` へ301リダイレクト | サービス一覧ページの表示 |
| `/services/*path` | `/projects` へ301リダイレクト | 削除（個別ルートで対応） |

### 3.3 維持するリダイレクト

旧サイトのURLからのアクセスに備えて以下のリダイレクトを追加する。

| パス | リダイレクト先 |
|------|---------------|
| `/contacts` | `/about`（既存のまま維持） |
| `/service` | `/services`（旧サイトURLの互換性） |
| `/service/development` | `/services/outsourcing` |
| `/service/consulting` | `/services/technical_advisor` |
| `/service/ses` | `/services`（リモートチームは終了済みのため一覧へ） |

## 4. ビジネスルール

### 4.1 サービスコンテンツ

- サービスの説明文は静的コンテンツとしてビューに直接記述する（データベース管理しない）
- コンテンツは旧サイト（rails-takeyuwebinc）の内容をベースに、実装フェーズで一人会社向けにリライトする
- リモートチームサービスは終了済みのため掲載しない

### 4.2 問い合わせ導線

- 問い合わせフォームは外部サービス（Google Forms）を使用する
- サイト内に自前の問い合わせフォームは設置しない
- Google FormsのURLは各ページで同一のものを使用する

## 5. SEO対応

### 5.1 メタ情報

各ページに以下のメタ情報を設定する。

| ページ | title | description |
|--------|-------|-------------|
| サービス一覧 | サービスの概要を示すタイトル | 提供サービスの概要 |
| 受託開発 | 受託開発サービスを示すタイトル | 受託開発サービスの概要 |
| 技術顧問 | 技術顧問サービスを示すタイトル | 技術顧問サービスの概要 |

### 5.2 サイトマップ

既存のサイトマップ生成処理に新規ページを追加する。

### 5.3 構造化データ

ホームページの既存構造化データ（Organization型）で `serviceType` が定義済みであるため、必要に応じてサービスページ側にも構造化データを追加する。
