# ADR 管理ワークフロー（配布 Skill v1.0.0）

このリポジトリの ADR は spotlight-rails の ADR 管理サーバーで一元管理する。
MCP ツール（`search_adrs_tool` / `get_adr_tool` / `register_adr_tool` /
`update_adr_tool` ほか）を通じて検索・参照・登録・更新を行う。

> **バージョン確認**: このスキルのバージョンは冒頭に表記されている。
> サーバー側のツール仕様と食い違う挙動があれば、古い配布 Skill を
> 使っている可能性をユーザーに報告すること。

## 前提設定

- **案件 code**: `<ENGAGEMENT_CODE>` ← このリポジトリの案件 code を記載する（配置時に必ず書き換えること）
- このリポジトリでの作業はすべて上記案件の文脈で行う。ADR の検索・登録時の
  `engagement_code` には原則この値を使う

## 標準ワークフロー

### 1. 案件の特定（初回・エラー時）

1. `find_adr_engagement_tool` に案件 code を渡して存在を確認する
2. 見つからない場合は `list_adr_engagements_tool` で**表記揺れがないか必ず確認**する
3. それでも存在しなければ、マスタを作成する（「マスタの作成」参照）

### 2. 実装・設計の前: 関連 ADR の検索と参照【必須】

実装・設計・技術選定に着手する前に、関連する過去の決定を必ず検索する。

**検索の必須手順**（埋め込み検索は語彙一致に強く、言い換えに弱い。
1回の検索で「関連 ADR なし」と結論づけてはならない）:

1. **複数の言い回しで自然言語検索を投げ、結果を統合する**
   - 例: 「デプロイ」の決定を探すなら「デプロイ方式」「リリース作業の手順」
     「本番反映のやり方」のように語彙を変えて 2〜3 回検索する
2. **キーワード・属性検索を併用する**
   - `keyword` に直接語彙（技術名・機能名）を渡して部分一致検索する
   - 必要に応じて `status: accepted` や日付で絞り込む
3. 他案件の前例を参考にしたい場合は `engagement_code` を外して**案件横断検索**する
4. 関連しそうな ADR は `get_adr_tool` で**全文を参照**し、以下を確認する:
   - 今回の実装が既存の決定（accepted）に反していないか
   - 同じ検討が過去に行われていないか（車輪の再発明の防止）
   - 置換変遷を確認し、参照した決定が最新（現行）かどうか

**決定に反する実装が必要になった場合**: 勝手に決定を無視せず、
置換指定付きの登録（後述）で決定自体を更新するか、ユーザーに相談する。

### 3. 決定を下したら: ADR の登録【必須】

アーキテクチャ上の決定（技術選定、設計方針、外部サービス連携方式、
データモデルの重要な変更など）を下したら `register_adr_tool` で登録する。

**記録ルール**:

- **逆ピラミッド**で書く: 結論（決定）を先に、経緯・詳細を後に
- **1ページ以内**に収める。長大な調査記録は ADR ではなく別文書にする
- タイトルは一覧で決定内容が判別できる一行にする（例: 「セッション保存先に Redis を採用」）
- 必須項目: `title` / `confidence`（高・中・低）/ `context`（現状の問題点・制約）/
  `decision`（採用した方針）/ `consequences`(ポジティブ・ネガティブ両方)
- `reevaluation_conditions`（再評価条件）をできる限り書く: どうなったら
  この決定を見直すべきかを、外部から判定可能な条件で記す
- `alternatives`: 検討して採用しなかった案と却下理由
- どのプロジェクト（期間）の決定か分かる場合は `project_name` を指定する

**ステータスの使い分け**:

| 場面 | status |
|------|--------|
| すでに下した決定の記録 | `accepted` |
| これから承認を得る提案 | `proposed` |
| 提案が承認された | `update_adr_tool` で `proposed → accepted` |
| 提案が不採用になった | `update_adr_tool` で `proposed → rejected` |
| 決定が後継なしで無効になった | `update_adr_tool` で `accepted → deprecated` |

### 4. 決定の変更: 置換指定付きの登録【必須】

既存の承認済み決定を置き換える場合は、**必ず** `register_adr_tool` の
`superseded_numbers` に旧 ADR の番号を指定して登録する。

- 新 ADR の登録・旧 ADR の「置換（superseded）」への変更・置換関係の記録が
  一体の操作として行われ、矛盾する承認済み ADR の並存を防げる
- ステータスを直接 `superseded` に変更することはできない（一体操作のみ）
- 置換対象は同一案件の `accepted` の ADR のみ。複数指定できる

## マスタの作成（クライアント・案件・プロジェクト）

新しい案件・プロジェクトの発生時は、人間のマスタ登録を待たずに MCP から作成してよい。

**誤作成の予防（必須手順）**:

1. 作成前に**必ず一覧ツール**（`list_adr_clients_tool` / `list_adr_engagements_tool` /
   `list_adr_projects_tool`）**で表記揺れがないか確認**する。
   類似 code・類似名称があればそれを使い、新規作成しない
2. code の命名規則: 小文字英数字とハイフンのみ（例: `takeyuweb`, `fabble`）。
   会社名・案件名を短くローマ字化 or 英語化したものを使う
3. クライアント code は全ドメイン共通の一意キー。工数管理で登録済みの
   取引先と同じ code を使えば自動的に同一実体として扱われる
4. 作成順序: クライアント → 案件 → プロジェクト（親が先）

## エラー時の振る舞い

- エラー応答には種別・原因パラメータ・推奨される次のアクションが含まれる。
  それに従って**自己解決してよい**（例: 案件が存在しない → 一覧で確認 → 作成）
- 検索が 0 件でも「関連 ADR なし」と即断せず、応答内のガイダンス
  （言い換え・キーワード切替・案件横断）に従って再検索する
- 自然言語検索が利用不可（search_unavailable）の場合はキーワード検索に切り替える
- 判断に迷う場合（決定の置換・削除・案件の変更など影響の大きい操作）は
  人間（ユーザー）に確認する
